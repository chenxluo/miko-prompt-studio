# 图像标注试验台软件总体设计文档

## 1. 项目定位

本工具定位为：

> 面向图像标注任务的本地交互式试验台，用于快速测试不同 API、模型配置、系统提示词、输出格式和图片输入组织方式下的标注效果与成本。

它首先是一个 **Prompt / API / 标注效果调试工具**，其次是一个 **小批量对比测试工具**，最后才是一个 **轻量批量运行辅助工具**。

它不应被设计成完整的数据生产平台或众包标注平台。
它的核心价值是帮助使用者快速回答：

```text
这张图 / 这组图，在这个 prompt、这个模型、这个参数、这个输出格式下，效果如何？成本是多少？是否值得扩大测试？
```

---

## 2. 核心目标

### 2.1 主要目标

1. 支持单图、少量图、多图组合、编辑前后图等图像标注任务调试。
2. 支持多平台视觉模型 API 调用。
3. 支持系统提示词、用户提示词、模型参数的快速修改和即时重跑。
4. 记录每次请求的输入、输出、token、成本、错误和人工备注。
5. 支持可编辑价格表，用于估算不同模型和任务规模下的成本。
6. 支持小批量样本测试，用于验证 prompt 稳定性。
7. 支持不同模型 / prompt / 参数配置的横向对比。
8. 支持自由文本、温和格式、宽松 JSON、严格 JSON 等不同输出模式。
9. 支持 CSV / JSONL / 文件夹 / 脚本式导入样本。
10. 为后续 agent 快速接入新平台 API 预留清晰扩展接口。

---

## 3. 非目标

第一阶段不追求：

1. 完整大规模生产调度系统。
2. 多用户权限系统。
3. 众包标注平台。
4. 完整数据集管理平台。
5. 高复杂度自动评测系统。
6. 对所有 provider 能力做完全统一抽象。
7. 对图片进行重度编辑、修复或生成。
8. 极致性能优化。

本工具主要面向本地、个人或小团队使用。
如果后续某个 prompt 和流程被验证稳定，大规模生产可以交给外部脚本或专门 pipeline 执行。

---

## 4. 基本设计原则

1. **Lab 优先**
   默认入口应是单图 / 少量图即时调试，而不是批量任务。

2. **Task 是沉淀结果，不是强制入口**
   用户可以先随手测试，满意后再保存为 Task、Prompt Version 或 Sample Set。

3. **image role 是软标签**
   图片角色用于组织 prompt 和显示，不作为硬编码枚举。

4. **不强制结构化输出**
   工具支持 JSON 解析和结构化显示，但不强制使用 strict JSON。

5. **raw response 永远保存**
   解析失败也不能丢失模型原始输出。

6. **成本统计是一等功能**
   每次请求都应记录 usage、价格快照和成本估算。

7. **原生 API adapter 优先**
   LiteLLM 可作为可选兜底，但核心 provider 应尽量原生适配。

8. **Python 保持核心业务逻辑**
   前端使用 TypeScript，但 API 调用、数据导入、成本计算、批量队列等核心逻辑优先放在 Python 后端。

9. **为 agent 扩展优化结构**
   新增平台、模型、导入方式或输出解析器时，应尽量只新增局部模块，不改动主流程。

---

## 5. 软件形态

推荐形态：

```text
Electron 桌面应用
  ├── Renderer: Web 前端
  ├── Main Process: Electron / Node
  └── Local Backend: Python 服务
```

### 5.1 为什么选择 Electron

本工具主要工作负载是：

* prompt 编辑
* 图片预览
* API 等待
* 结果展示
* 表格统计
* JSON / markdown 显示
* 历史记录管理

它不是高性能图形软件，因此 Electron 的性能开销可以接受。
Electron 的优势是：

1. 前端生态成熟。
2. Monaco Editor、表格、JSON Viewer、Markdown Viewer、Diff Viewer 等组件丰富。
3. 打包桌面应用成熟。
4. 文件系统访问方便。
5. agent 后续修改和扩展更容易。
6. 与 Python 后端组合自然。

## 8. 主要功能模块

## 8.1 Lab

Lab 是主入口。

目标是支持用户快速完成：

```text
拖入图片 → 写 prompt → 选模型 → Run → 看结果 → 改 prompt → 再 Run
```

### Lab 需要支持

1. 单图输入。
2. 多图输入。
3. 图片 role 软标签。
4. system prompt 编辑。
5. user prompt 编辑。
6. prompt template 变量预览。
7. 模型选择。
8. 参数调整。
9. 输出格式模式选择。
10. 图片预处理策略选择。
11. 单次运行。
12. 多模型并行运行。
13. 运行历史对比。
14. token / 成本即时显示。
15. 将当前配置保存为 Prompt Version / Task / Model Config。

### Lab 的核心体验

Lab 不应强迫用户先创建 Task。
用户可以直接开始试验。

每次点击 Run 都应产生一条可追溯 Run Record。

---

## 8.2 Prompt Library

Prompt Library 用于保存和版本化提示词。

### Prompt 应拆分为

1. system prompt
2. user prompt template
3. format instruction
4. optional few-shot examples
5. notes

Prompt Version 应记录：

```text
prompt_id
prompt_version_id
name
created_at
system_prompt
user_template
format_instruction
notes
```

用户在 Lab 中调整 prompt 后，可以：

1. 临时运行，不保存。
2. 保存为当前 prompt 的新版本。
3. 另存为新 prompt。
4. 与旧版本 diff。

---

## 8.3 Model Configs

Model Config 用于保存常用模型参数组合。

字段示例：

```text
provider_id
model_id
adapter_id
temperature
max_output_tokens
top_p
seed
provider_options
notes
```

同一 provider / model 可以保存多个配置，例如：

```text
qwen-vl-max / precise-json
qwen-vl-max / creative-caption
gemini / long-detailed-caption
```

---

## 8.4 Samples / Sample Sets

Samples 用于组织测试样本。

本工具不需要成为完整数据集管理平台，但需要支持轻量样本组织。

### 支持来源

1. 手动拖入图片。
2. CSV / TSV。
3. JSONL manifest。
4. 文件夹扫描。
5. Python import script。

### Sample Set 典型用途

1. golden set。
2. 某类任务的小测试集。
3. 难例集。
4. 对比测试集。
5. 最近导入的一批样本。

---

## 8.5 Batch Test

Batch Test 是 Lab 的扩展，不是主入口。

典型流程：

```text
当前 prompt 在单图上效果不错
  ↓
选择 Run on Sample Set
  ↓
选择 CSV / JSONL / Sample Set
  ↓
预览前几条
  ↓
估算成本
  ↓
先跑 10 / 50 / 全部
  ↓
查看结果和失败项
```

### Batch 必须支持

1. 先跑 N 条。
2. 运行前成本估算。
3. 取消运行。
4. 失败重跑。
5. 跳过已完成项。
6. 实时显示成功数、失败数、成本。
7. 导出结果。

### Batch 不必第一阶段支持

1. 分布式调度。
2. 复杂优先级队列。
3. 多机器执行。
4. 生产级任务监控。

---

## 8.6 Compare

Compare 用于横向比较。

比较维度包括：

1. 同一 prompt，不同模型。
2. 同一模型，不同 prompt。
3. 同一模型和 prompt，不同参数。
4. 同一配置，不同输出格式约定。
5. 同一配置，不同图片预处理策略。

Compare 本质是矩阵运行：

```text
samples × prompts × model_configs
```

结果展示应支持：

1. 按样本横向看多个输出。
2. 按模型聚合看平均成本。
3. 按 prompt 聚合看通过率。
4. 标记哪个结果更好。
5. 快速保存优胜配置。

---

## 8.7 Pricing

Pricing 是核心模块。

### Pricing Profile

每个模型可以有一个或多个价格配置。

字段示例：

```text
provider_id
model_id
currency
effective_date
input_token_price
output_token_price
image_price_mode
image_token_price
image_per_request_price
cached_input_price
batch_discount
notes
```

价格必须可编辑。
每次 Run 必须保存价格快照，避免未来价格变化导致历史成本不一致。

### Cost Estimation

工具需要支持：

1. 单次请求成本。
2. 当前 Lab 历史成本。
3. Batch 预计成本。
4. 已运行成本。
5. 重试额外成本。
6. 按输入、输出、图片拆分成本。

---

## 8.8 Output Viewer

输出显示不应假设都是 JSON。

支持模式：

1. raw text
2. markdown
3. soft sections
4. loose JSON
5. strict JSON
6. parsed table
7. diff view

### Output Contract

输出模式分为：

```text
free_text
soft_sections
loose_json
strict_json
custom
```

默认推荐 `free_text` 或 `soft_sections`。
`strict_json` 只在任务确实需要时启用。

---

## 8.9 Review

Review 是轻量人工评估，不是强制 rubric。

每个 Run Item 可以记录：

```text
accepted
rating
labels
notes
reviewed_at
```

内置标签示例：

```text
good
bad
needs_revision
refusal
hallucination
missing_detail
format_error
too_verbose
too_short
wrong_image_order
wrong_role_reference
safety_overrefusal
```

用户可以自定义标签。

Review 的作用：

1. 辅助人工调试。
2. 统计小批量结果质量。
3. 标记可导出结果。
4. 形成 hard cases。

---

## 8.10 Export

导出不是第一核心，但需要基础支持。

支持导出：

1. JSONL。
2. CSV。
3. 图片路径 + caption。
4. source / target / instruction 编辑任务格式。
5. raw response。
6. parsed response。
7. accepted-only results。
8. failed items。

导出时应包含必要追溯信息：

```text
sample_id
prompt_version_id
model_id
run_id
created_at
cost
review_status
```

## 9. API Provider 设计

## 9.1 基本策略

采用：

```text
原生 Provider Adapter 优先
Generic OpenAI-compatible Adapter 兜底
LiteLLM Adapter 可选
```

不把 LiteLLM 作为核心依赖。

原因：

1. 多图格式差异大。
2. usage 返回差异大。
3. 图片计费方式差异大。
4. safety settings 差异大。
5. structured output 支持差异大。
6. batch API 差异大。
7. 本工具需要掌握 provider-specific 细节。

---

## 9.2 Provider Adapter 职责

每个 adapter 负责：

```text
list_models()
get_capabilities()
build_provider_request()
send()
parse_response()
parse_usage()
normalize_error()
```

Adapter 输入是 Internal Request。
Adapter 不直接读取 UI 状态，不直接读数据库，不直接处理 Sample Record。

---

## 9.3 Capability 描述

每个模型需要能力描述：

```text
supports_image
supports_multi_image
supports_system_prompt
supports_json_mode
supports_strict_json_schema
supports_batch_api
max_images
max_output_tokens
known_quirks
```

known_quirks 很重要，例如：

```text
bbox 强但安全拒绝多
多图容易混淆顺序
JSON 稳定但细节少
中文输出较好
输出偏短
```

---

## 9.4 新增平台的理想流程

设计目标是让 agent 后续可以快速接入平台。

新增 provider 应主要涉及：

```text
1. 新增 adapter 文件
2. 新增 capability config
3. 新增 pricing profile
4. 新增少量测试样例
```

主流程不应需要重构。

## 10. 批量输入设计

## 10.1 标准中间格式

所有批量输入最终转换为 Sample Record。

支持三层导入：

```text
Level 1: CSV / TSV + 列映射
Level 2: JSONL manifest
Level 3: Python import script
```

---

## 10.2 CSV / TSV

适合简单任务。

示例：

```csv
id,image,extra_prompt,category
0001,images/0001.png,请重点描述服装和动作,pose
0002,images/0002.png,请重点描述场景和光照,scene
```

导入时通过列映射配置：

```text
id column       → sample_id
image column    → image role
text column     → vars
other column    → metadata
```

---

## 10.3 JSONL Manifest

适合复杂多图任务。

示例：

```json
{"sample_id":"001","images":[{"role":"source","path":"before/001.png"},{"role":"target","path":"after/001.png"}],"vars":{"edit_instruction":"change outfit"}}
```

---

## 10.4 Python Import Script

适合高级数据组织。

脚本职责：

```text
任意输入 → Sample Records
```

脚本不直接控制运行流程。

可用于：

1. before / after 自动配对。
2. 连续帧自动成组。
3. 从 txt 读取 caption。
4. 从 metadata 读取变量。
5. 自动选择 reference image。
6. 根据文件名生成 prompt 变量。

---

## 11. 图片处理策略

图片预处理应是显式配置，而不是隐藏行为。

支持策略：

```text
original
resize_long_edge
resize_short_edge
fit_within_box
center_crop
convert_format
jpeg_quality
png
```

每次 Run 需要记录：

1. 原始路径。
2. 原始尺寸。
3. 请求尺寸。
4. 请求格式。
5. 文件大小。
6. sha256。
7. 预处理策略。

这对图像 token 成本、bbox 坐标和结果复现很重要。

### 12.1 主要表

## 13. API Key 管理

API key 不应明文混在普通导出配置中。

推荐：

1. 使用系统 keychain。
2. 简化版可使用本地加密配置文件。
3. Run Record 中不保存 API key。
4. 导出配置时默认不包含 key。
5. UI 中显示 key 时应脱敏。

## 16. 推荐迭代路线

### Phase 1: 单图 Lab 可用

目标：

```text
能拖入图片，写 prompt，调用一个 API，看到输出和成本。
```

内容：

1. Electron + Python 后端打通。
2. Lab 基础 UI。
3. Prompt 编辑器。
4. 图片预览。
5. 一个 provider adapter。
6. Run Record 保存。
7. Pricing Profile 基础成本计算。

---

### Phase 2: 多图与历史对比

目标：


## 18. 风险与注意点

### 18.1 不要过早平台化

MVP 不要做复杂插件市场。
先保留 adapter interface，让实现层加平台容易即可。

### 18.2 不要强制 JSON

很多图像标注任务需要模型自由表达。
JSON 支持应作为能力，而不是默认路径。

### 18.3 不要让 batch 压过 Lab

本工具核心是调试与试验。
batch 是扩大验证，不是主流程。

### 18.4 不要把数据集管理做太重

Sample Set 只需支撑测试样本和小批量实验。
完整数据资产管理可以交给外部系统。

### 18.5 不要丢失历史快照

Prompt、model config、pricing 都会变。
Run Record 必须保存运行时快照。

### 18.6 不要隐藏图片预处理

图片尺寸和压缩会影响质量与成本。
必须记录实际请求图片。

## 20. 最终设计摘要

本工具应设计为：

```text
Electron 桌面应用
+ React / TypeScript 前端
+ Python FastAPI 后端
+ SQLite 本地数据库
+ 原生 Provider Adapter
+ 可编辑 Pricing Profile
+ Lab 优先的图像标注试验台
```

核心工作方式：

```text
单图 / 少量图即时调试
  ↓
保存 prompt / model config
  ↓
小批量 sample set 测试
  ↓
横向比较模型和 prompt
  ↓
导出可用结果或交给外部生产脚本
```

核心边界：

```text
它不是完整生产平台。
它不是普通聊天 playground。
它是图像标注任务的 prompt、API、成本和结果质量试验台。
```
