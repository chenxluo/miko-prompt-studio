# 图像标注试验台核心数据结构设计草案

## 0. 总体目标

本工具的核心流程可以抽象为：

```text
Sample Record
    ↓
Prompt / Model / Output / Preprocess 配置
    ↓
Internal Request
    ↓
Provider Adapter
    ↓
Raw Response
    ↓
Run Record
```

三类核心对象的职责如下：

| 对象               | 负责什么                                 | 不负责什么               |
| ---------------- | ------------------------------------ | ------------------- |
| Sample Record    | 描述样本本身：图片、变量、metadata                | 不描述具体模型、prompt、运行结果 |
| Internal Request | 描述一次即将发给模型的标准化请求                     | 不绑定某个平台 API 的原始格式   |
| Run Record       | 描述一次运行发生了什么：输入快照、输出、usage、成本、错误、人工备注 | 不作为样本源数据本身          |

设计原则：

1. **Sample Record 是数据层。**
2. **Internal Request 是执行前的标准中间表示。**
3. **Run Record 是执行后的审计记录。**
4. **Provider Adapter 只负责 Internal Request ↔ 平台 API 的转换。**
5. **raw response 永远保存。**
6. **prompt、model config、pricing profile 在 Run 中保存快照，保证可复现。**
7. **image role 是软标签，不是硬 schema。**
8. **结构化输出是可选能力，不是强制路径。**

---

# 1. Sample Record

## 1.1 定位

Sample Record 描述一条待测试或待标注样本。

它可以来自：

* 单图拖拽
* 多图手动组合
* CSV / TSV
* JSONL manifest
* 文件夹扫描
* Python import script
* 未来的数据集系统

无论来源如何，进入工具后都统一转换为 Sample Record。

---

## 1.2 Sample Record 的核心结构

推荐基础格式：

```json
{
  "schema_version": "sample_record.v1",
  "sample_id": "sample_000001",
  "sample_type": "single_image",
  "images": [
    {
      "image_id": "img_0001",
      "role": "target",
      "path": "images/000001.png",
      "uri": null,
      "mime_type": "image/png",
      "display_name": "000001.png",
      "order": 0,
      "metadata": {
        "width": 1024,
        "height": 1536,
        "file_size": 1839201,
        "sha256": "..."
      }
    }
  ],
  "vars": {
    "extra_prompt": "请重点描述人物动作、服装和场景。",
    "language": "zh"
  },
  "metadata": {
    "category": "pose",
    "difficulty": "medium",
    "source": "manual_import"
  },
  "expected": null,
  "tags": ["test", "golden_candidate"],
  "notes": ""
}
```

---

## 1.3 字段说明

### `schema_version`

用于未来迁移。

```json
"schema_version": "sample_record.v1"
```

不要假设所有历史样本永远使用同一结构。

---

### `sample_id`

样本的稳定 ID。

要求：

* 在一个 Sample Set 内唯一。
* 尽量稳定，不要每次导入都变化。
* 可来自文件名、CSV id 列、hash 或用户指定。

示例：

```json
"sample_id": "edit_000123"
```

---

### `sample_type`

表示样本大类，但不应限制过死。

建议内置值：

```text
single_image
multi_image
edit_pair
image_group
custom
```

说明：

* `single_image`：一张目标图。
* `multi_image`：多张图共同输入。
* `edit_pair`：编辑前 / 编辑后 / 指令。
* `image_group`：一组连续帧、一组候选图、一组参考图。
* `custom`：暂未归类的复杂样本。

`sample_type` 只用于 UI 默认展示和导入提示，不应作为硬约束。

---

### `images`

图片引用列表。

每张图片是一条 `ImageRef`。

```json
{
  "image_id": "img_0001",
  "role": "target",
  "path": "images/000001.png",
  "uri": null,
  "mime_type": "image/png",
  "display_name": "目标图",
  "order": 0,
  "metadata": {
    "width": 1024,
    "height": 1536,
    "file_size": 1839201,
    "sha256": "..."
  }
}
```

#### image role 设计

`role` 是软标签，可任意扩展。

推荐预设：

```text
target
source
before
after
reference
character_reference
style_reference
pose_reference
mask
control_image
candidate
frame
```

但用户可以自定义：

```text
角色设定图
场景参考
错误案例
修正结果
```

系统不应要求某种 sample_type 必须包含某些 role。
例如 `edit_pair` 通常有 `source` 和 `target`，但不强制。

---

### `vars`

进入 prompt template 的可变字段。

例如：

```json
"vars": {
  "extra_prompt": "请重点描述姿势。",
  "edit_instruction": "把人物衣服改成白色连衣裙。",
  "character_name": "girl_A",
  "output_language": "zh"
}
```

建议支持值类型：

```text
string
number
boolean
array
object
null
```

但在 prompt 渲染时，复杂对象需要有明确序列化策略。

推荐模板引用方式：

```text
{{vars.extra_prompt}}
{{vars.edit_instruction}}
{{sample.sample_id}}
{{metadata.category}}
```

不建议一开始允许任意 Python 表达式插值，避免模板系统变复杂。

---

### `metadata`

不一定进入 prompt 的辅助信息。

例如：

```json
"metadata": {
  "category": "nsfw_pose",
  "difficulty": "hard",
  "source_dataset": "internal_test_set",
  "original_caption": "..."
}
```

用途：

* 过滤样本
* 分组统计
* 构建 batch
* 导出
* 调试备注
* 可选进入 prompt

---

### `expected`

可选的参考答案或人工标注。

例如：

```json
"expected": {
  "caption": "一名女性坐在椅子上，穿白色连衣裙。",
  "tags": ["sitting", "white dress"],
  "bbox": []
}
```

用途：

* golden set
* 自动对比
* 人工检查
* 后续 judge model

MVP 可以不重点实现，但字段应预留。

---

### `tags`

样本级标签。

```json
"tags": ["golden", "hard_case", "multi_person"]
```

用于快速筛选。

---

### `notes`

人工备注。

```json
"notes": "这个样本主要测试模型是否能识别手部错误。"
```

---

## 1.4 Sample Set

单条 Sample Record 通常归属于一个 Sample Set。

Sample Set 用于组织一批样本。

```json
{
  "schema_version": "sample_set.v1",
  "sample_set_id": "golden_pose_001",
  "name": "姿势标注 Golden Set",
  "description": "用于测试姿势、服装、场景描述能力的小批量样本。",
  "created_at": "2026-06-21T12:00:00+09:00",
  "import_source": {
    "type": "csv",
    "path": "imports/pose_test.csv"
  },
  "records": [
    "sample_000001",
    "sample_000002"
  ],
  "metadata": {
    "default_task": "pose_caption"
  }
}
```

Sample Set 本身不直接包含图片数据，只引用 Sample Record。

---

## 1.5 CSV 导入映射

CSV 不直接等于 Sample Record。
CSV 需要通过列映射转换。

示例 CSV：

```csv
id,target_image,extra_prompt,category
0001,images/0001.png,请重点描述服装和动作,pose
0002,images/0002.png,请重点描述场景和光照,scene
```

映射配置：

```json
{
  "id_column": "id",
  "image_columns": [
    {
      "column": "target_image",
      "role": "target"
    }
  ],
  "var_columns": [
    "extra_prompt"
  ],
  "metadata_columns": [
    "category"
  ]
}
```

转换结果：

```json
{
  "sample_id": "0001",
  "sample_type": "single_image",
  "images": [
    {
      "role": "target",
      "path": "images/0001.png"
    }
  ],
  "vars": {
    "extra_prompt": "请重点描述服装和动作"
  },
  "metadata": {
    "category": "pose"
  }
}
```

---

## 1.6 JSONL manifest

复杂样本推荐直接使用 JSONL。

每行是一条 Sample Record 或接近 Sample Record 的简化格式。

示例：

```json
{"sample_id":"001","sample_type":"edit_pair","images":[{"role":"source","path":"before/001.png"},{"role":"target","path":"after/001.png"}],"vars":{"edit_instruction":"change outfit"}}
{"sample_id":"002","sample_type":"multi_image","images":[{"role":"target","path":"target/002.png"},{"role":"character_reference","path":"ref/002_a.png"},{"role":"pose_reference","path":"ref/002_b.png"}],"vars":{"extra_prompt":"判断目标图是否保持角色一致"}}
```

---

## 1.7 脚本式导入

脚本式导入不直接控制运行，只负责生成 Sample Record。

推荐约定：

```text
input: 任意目录 / 文件 / metadata
output: JSONL Sample Records
```

这样可以支持：

* 文件夹自动配对 before / after
* 连续帧自动成组
* 从 `.txt` caption 文件读取变量
* 从 metadata 判断 SFW / NSFW
* 自动抽取 reference image

脚本导入的输出仍然进入统一校验流程。

---

# 2. Internal Request

## 2.1 定位

Internal Request 是一次模型请求的 provider-independent 标准表示。

它由以下部分共同生成：

```text
Sample Record
+ Prompt Spec
+ Model Config
+ Output Contract
+ Image Preprocess Config
+ Runtime Options
= Internal Request
```

Internal Request 不等于 OpenAI request、Gemini request、Qwen request。
它是工具内部的标准中间层。

Provider Adapter 负责将 Internal Request 转换成具体平台 API。

---

## 2.2 Internal Request 基础结构

```json
{
  "schema_version": "internal_request.v1",
  "request_id": "req_20260621_000001",
  "sample_ref": {

    "render_context": {
      "vars": {
        "extra_prompt": "请重点描述服装和动作"
      },
      "metadata": {
        "category": "pose"
      }
    },
    "template_refs": {
      "prompt_id": "prompt_pose_caption",
      "prompt_version_id": "prompt_pose_caption_v7"
    }
  },
  "images": [
    {
      "request_image_id": "rimg_0001",
      "source_image_id": "img_0001",
      "role": "target",
      "path": "images/000001.png",
      "mime_type": "image/png",
      "order": 0,

```

---

## 2.3 Prompt 部分

### `system_prompt`

完整渲染后的 system prompt。

```json
"system_prompt": "你是图像标注助手。"
```

如果某个平台不支持 system prompt，由 adapter 决定如何降级，例如合并进 user prompt。

---

### `user_prompt`

完整渲染后的 user prompt。

```json
"user_prompt": "请为这张图生成训练用标注。\n\n额外要求：请重点描述服装和动作。"
```

Internal Request 中保存的是渲染后的最终文本。
这样 Run Record 可以复现当时实际发出的语义。

---

### `render_context`

保存模板渲染时用到的变量快照。

```json
"render_context": {
  "vars": {
    "extra_prompt": "请重点描述服装和动作"
  },
  "metadata": {
    "category": "pose"
  }
}
```

这样可以追溯 prompt 里某段文本来自哪里。

---

### `template_refs`

引用原始 prompt 版本。

```json
"template_refs": {
  "prompt_id": "prompt_pose_caption",
  "prompt_version_id": "prompt_pose_caption_v7"
}
```

注意：Run Record 中应同时保存引用和快照。
即使 prompt_v7 以后被编辑，历史 run 仍然可复现。

---

## 2.4 Images 部分

Internal Request 的 images 是“本次请求实际会使用的图片列表”。

它不是 Sample Record 里的原始图片列表的简单拷贝，因为它可能经过：

* resize
* crop
* format conversion
* compression
* orientation fix
* base64 conversion
* 临时缓存
* 过滤某些 role

每张图包含：

```json
{
  "request_image_id": "rimg_0001",
  "source_image_id": "img_0001",
  "role": "target",
  "path": "images/000001.png",
  "mime_type": "image/png",
  "order": 0,
  "preprocess": {
    "mode": "resize_long_edge",
    "long_edge": 1024,
    "format": "jpeg",
    "quality": 92
  },
  "resolved": {
    "path": "cache/preprocessed/req_000001_img_0001.jpg",
    "mime_type": "image/jpeg",
    "width": 682,
    "height": 1024,
    "file_size": 312901,
    "sha256": "..."
  }
}
```

### `role`

仍然是软标签。

Adapter 可以根据 role 组织最终请求，例如在 user prompt 中插入：

```text
Image 1: target
Image 2: reference
```

但 role 本身不强制绑定特定 API 能力。

---

### `order`

多图请求中图片顺序很重要。

`order` 表示图片送入模型的顺序。
如果 provider API 按内容数组顺序解释图片，adapter 必须遵守这个顺序。

---

### `preprocess`

记录预处理策略。

```json
"preprocess": {
  "mode": "resize_long_edge",
  "long_edge": 1024,
  "format": "jpeg",
  "quality": 92
}
```

---

### `resolved`

记录实际请求使用的图片版本。

```json
"resolved": {
  "path": "cache/preprocessed/req_000001_img_0001.jpg",
  "mime_type": "image/jpeg",
  "width": 682,
  "height": 1024,
  "file_size": 312901,
  "sha256": "..."
}
```

这对成本估算、bbox 坐标、质量回溯非常重要。

---

## 2.5 Model 部分

```json
"model": {
  "provider_id": "gemini",
  "model_id": "gemini-xxx",
  "adapter_id": "native_gemini",
  "parameters": {
    "temperature": 0.2,
    "max_output_tokens": 2048,
    "top_p": null,
    "seed": null
  },
  "provider_options": {
    "safety_settings": "default"
  }
}
```

### 通用参数

建议第一版支持：

```text
temperature
max_output_tokens
top_p
seed
stop
```

不是所有 provider 都支持所有参数。
Adapter 应负责：

* 支持则传递
* 不支持则忽略或报 warning
* 参数名转换

---

### provider_options

用于平台特殊能力。

例如：

```json
"provider_options": {
  "safety_settings": "less_restrictive",
  "response_mime_type": "application/json"
}
```

这部分不应该进入通用参数，否则核心 schema 会不断膨胀。

---

## 2.6 Output Contract

Output Contract 描述期望输出形式，但不强制必须是结构化。

建议模式：

```text
free_text
soft_sections
loose_json
strict_json
custom
```

示例：

```json
{
  "mode": "soft_sections",
  "format_instruction": "请按主体、动作、服装、场景、质量问题几个小标题输出。",
  "json_schema": null,
  "parser": {
    "type": "section_parser",
    "options": {
      "section_names": ["主体", "动作", "服装", "场景", "质量问题"]
    }
  }
}
```

### free_text

不要求格式。

```json
{
  "mode": "free_text",
  "format_instruction": null,
  "parser": null
}
```

### soft_sections

温和格式。

```json
{
  "mode": "soft_sections",
  "format_instruction": "请按主体、动作、服装、场景、画风输出。",
  "parser": {
    "type": "section_parser"
  }
}
```

### loose_json

提示模型输出 JSON，但不启用强约束。

```json
{
  "mode": "loose_json",
  "format_instruction": "请输出 JSON，不要添加解释。",
  "json_schema": {
    "type": "object",
    "properties": {
      "caption": {"type": "string"},
      "tags": {"type": "array"}
    }
  },
  "parser": {
    "type": "json_repair_parser"
  }
}
```

### strict_json

启用平台结构化输出能力，如果 provider 支持。

```json
{
  "mode": "strict_json",
  "json_schema": {
    "type": "object",
    "required": ["caption"],
    "properties": {
      "caption": {"type": "string"}
    }
  },
  "parser": {
    "type": "strict_json_parser"
  }
}
```

Adapter 需要根据 provider capability 判断是否支持 strict mode。
不支持时应提示，而不是静默降级。

---

## 2.7 Runtime 部分

```json
"runtime": {
  "timeout_seconds": 120,
  "retry_policy": {
    "max_retries": 1,
    "retry_on": ["rate_limit", "timeout", "format_error"]
  },
  "dry_run": false
}
```

`dry_run` 用于只估算成本、不实际请求。

---

## 2.8 Internal Request 的生命周期

```text
1. 用户在 Lab 或 Batch 中选择样本
2. 选择 prompt version
3. 选择 model config
4. 选择 output contract
5. 应用图片预处理策略
6. 渲染 prompt template
7. 生成 Internal Request
8. 校验 Internal Request
9. 交给 Provider Adapter
10. Adapter 转换成平台原始请求
```

校验点：

* 图片路径是否存在
* prompt 变量是否缺失
* 图片数量是否超过模型上限
* output contract 是否被该模型支持
* max tokens 是否超过模型限制
* pricing profile 是否存在
* provider key 是否已配置

---

# 3. Run Record

## 3.1 定位

Run Record 是执行记录。

它要回答几个问题：

1. 这次运行用了哪些样本？
2. 用了哪个 prompt？
3. 用了哪个模型配置？
4. 实际向 API 发送了什么？
5. 返回了什么？
6. 成功还是失败？
7. 消耗了多少 token / 图片 / 金额？
8. 人工觉得结果如何？
9. 如果重试，重试了几次？
10. 以后能否复现或导出？

---

## 3.2 分层结构

推荐把 Run Record 分成三层：

```text
Run Session
  └── Run Item
        └── Attempt
```

### Run Session

一次整体运行。

例如：

* Lab 单图运行一次
* 对 20 张图跑一个 prompt
* 对 20 张图 × 3 个模型做 compare
* batch 测试前 100 条

### Run Item

Run Session 中的一条样本结果。

例如 sample_0001 在 prompt_v7 + qwen 下的结果。

### Attempt

一次实际 API 请求。

如果超时重试、格式错误重试，则一个 Run Item 下会有多个 Attempt。

---

## 3.3 Run Session 结构

```json
{
  "schema_version": "run_session.v1",
  "run_id": "run_20260621_000001",
  "run_type": "lab",
  "name": "单图姿势标注测试",
  "status": "completed",
  "created_at": "2026-06-21T12:00:00+09:00",
  "started_at": "2026-06-21T12:00:03+09:00",
  "completed_at": "2026-06-21T12:00:12+09:00",
  "source": {
    "mode": "lab",
    "sample_set_id": null,
    "sample_ids": ["sample_000001"]
  },
  "config_snapshot": {
    "prompt_version": {
      "prompt_id": "prompt_pose_caption",
```

## 3.8 Internal Request Snapshot

Run Item 应保存完整 Internal Request 快照，而不仅是引用。

原因：

* prompt 可能后来被改。
* model config 可能后来被改。
* pricing profile 可能后来被改。
* 图片预处理策略可能后来被改。
* sample metadata 可能后来被改。

为了可复现，Run Item 至少要保存：

```text
渲染后的 system prompt
渲染后的 user prompt
使用的图片路径和预处理结果
模型参数
output contract
pricing profile id
```

可以不在 Run Item 表中直接存大 JSON，而是存到单独的 `internal_request_snapshots` 表或文件。

## 3.11 Error 结构

失败时保存规范化错误。

```json

  "provider_error_code": "429",
  "retryable": true,
  "raw_error": {}
```

推荐错误类型：

```text
auth_error
rate_limit
timeout
network_error
provider_error
invalid_request
unsupported_capability
safety_blocked
empty_response
parse_error
format_error
unknown_error
```

注意：

* `parse_error` 不一定代表 API 调用失败。
* 如果 raw response 成功返回，但 JSON 解析失败，Attempt 可以是 `succeeded`，Run Item 可以根据策略标记为 `failed` 或 `succeeded_with_parse_error`。
* 可以在 Run Item 层增加 `quality_status`，避免 status 过载。

## 3.12 Response 结构

Run Item 中的 response 应分三层：

```json
"response": {
  "raw_text": "...",
  "parsed": {},
  "parse_status": "parsed",
  "parse_errors": []
}
```

### raw_text

模型原始文本输出。永远保存。

### parsed

工具根据 output contract 解析出的结构。

可能是：

* JSON object
* sections object
* tag list
* bbox list
* null

### parse_status

```text
not_parsed
parsed
partially_parsed
parse_failed
```

### parse_errors

例如：

```json
[
  {
    "type": "missing_section",
    "message": "未找到「质量问题」小标题。"
  }
]
```

# 4. 三者之间的关系

## 4.1 从 Sample Record 到 Internal Request

```text
Sample Record:
  images: target image
  vars: extra_prompt

Prompt Template:
  "请描述图片。额外要求：{{vars.extra_prompt}}"

Model Config:
  qwen-vl-max, temperature=0.2

Output Contract:
  soft_sections

Preprocess:
  resize long edge 1024

生成：

Internal Request:
  rendered prompt
  resolved image
  model config
  output contract
```

---

## 4.2 从 Internal Request 到 Run Record

```text
Internal Request
  ↓
Provider Adapter
  ↓
Provider Raw Request
  ↓
API Call
  ↓
Raw Response / Error
  ↓
Normalize Response
  ↓
Parse Response
  ↓
Estimate Cost
  ↓
Run Item + Attempt
```

---

## 4.3 Lab 模式

Lab 模式中，通常只有一个或几个 Sample Record。

特点：

* 用户频繁修改 prompt 和模型参数。
* 每点一次 Run，生成一个 Run Session 或追加到当前 Lab Session。
* 每次 Run 都保存 Internal Request 快照。
* 用户可以对比历史输出。

推荐：

```text
一个 Lab 页面可以对应一个 Lab Session。
每次点击 Run 生成一个 Run Item。
如果同时跑多个模型，则生成多个 Run Item。
```

---

## 4.4 Batch 模式

Batch 模式中，一个 Run Session 包含多个 Run Item。

```text
Run Session: batch_001
  Run Item: sample_001
  Run Item: sample_002
  Run Item: sample_003
```

每个 Run Item 有自己的 Internal Request，因为每条样本的 vars 和图片不同。

---

## 4.5 Compare 模式

Compare 模式本质上是矩阵运行。

例如：

```text
20 samples × 3 prompts × 2 models = 120 run items
```

Run Item 应额外记录维度：

```json
"compare_axes": {
  "prompt_version_id": "prompt_v7",
  "model_config_id": "qwen_config_01"
}
```

这样前端可以做横向表格。

---

# 5. Provider Adapter 相关约定

## 5.1 Adapter 输入

Adapter 只接收 Internal Request。

```text
Internal Request → build_provider_request()
```

Adapter 不应该直接读取 Sample Record、Prompt 表、Pricing 表。
这些都应该在生成 Internal Request 前完成。

---

## 5.2 Adapter 输出

Adapter 返回标准化 Attempt Result。

```json
{
  "status": "succeeded",
  "provider_response_raw": {},
  "normalized_response": {
    "text": "...",
    "finish_reason": "stop",
    "safety": {}
  },
  "usage": {},
  "error": null,
  "latency_ms": 8123
}
```

---

## 5.3 Adapter Capability

每个 adapter / model 应有 capability 描述。

```json
{
  "provider_id": "qwen",
  "model_id": "qwen-vl-max",
  "supports_image": true,
  "supports_multi_image": true,
  "supports_system_prompt": true,
  "supports_json_mode": false,
  "supports_strict_json_schema": false,
  "supports_batch_api": false,
  "max_images": 20,
  "max_output_tokens": 8192,
  "notes": "bbox 表现较好，但安全拒绝可能较多。"
}
```

生成 Internal Request 时先检查 capability。

# 9. 核心边界总结

最终边界应保持清晰：

```text
Sample Record:
  样本是什么。

Internal Request:
  这一次要如何请求模型。

Run Record:
  这一次实际发生了什么。
```

不要让 Sample Record 存 prompt。
不要让 Internal Request 依赖 UI 状态。
不要让 Provider Adapter 直接操作数据库。
不要只保存 parsed response 而丢掉 raw response。
不要把 image role 做成硬编码枚举。
不要把 strict JSON 作为默认输出路线。

这三个对象稳定后，后续无论是 Electron 前端、Python 后端、CSV 导入、脚本导入、原生 API adapter、LiteLLM adapter、batch 测试、compare 测试，都可以围绕这套核心结构扩展。
